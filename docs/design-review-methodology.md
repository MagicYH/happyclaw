# 设计评审方法论：横切维度与用户旅程走查

> 本文档源于一次真实评审复盘：review 某爽约黑名单后端方案时漏掉了「黑名单入库无灰度、但 Bot 会向非灰度城市用户发送」这一跨模块一致性缺陷。复盘结论沉淀为以下方法论，供后续所有技术方案评审复用。

---

## 一、为什么会漏

| 原因 | 具体表现 |
|---|---|
| 按章节读，没做横切审查 | 顺着文档 §1 → §2 → §3 阅读，每节在自身上下文里都自洽就放过；不回头按「维度」做交叉一致性检查 |
| 心智模型建在「表 + 任务」而非「用户感知」 | 脑子里画的是「哪个 Job 写哪张表」，而不是「一个非灰度城市用户端到端会收到/被拦截什么」 |
| 看到了决策点但只想一层 | 已注意到「全量记账」决策，只想到跨城爽约的尴尬，没追问「下游消费它的每个逻辑是否都按灰度过滤」 |
| 方案结构天然隐藏沉默项 | 灰度判断显式出现在 §拦截 和 §取消限制；§Bot 通知一节完全没提「灰度」→ 沉默 = 默认全量 → 不会让读者皱眉 |

> **核心教训**：review 时最危险的不是方案里写错的地方，而是方案里**没写**的地方。

---

## 二、评审 5 条硬规则

### Rule 1 — 横切维度矩阵（必做）

读完详细设计后，**强制**列一张 feature × dimension 的矩阵，每格打钩或标「?」。对业务后端方案至少覆盖以下维度：

- 灰度 / 开关
- 幂等性
- 事务边界
- 并发安全
- 权限校验
- 异常处理 / 降级
- 可观测性（日志 / 监控 / 告警）
- i18n / 多地域

示例（爽约黑名单）：

| 维度 | 记账 | 拦截 | 警告 Bot | 惩罚 Bot | 取消限制 | 定时递补 | 后台查询 |
|---|---|---|---|---|---|---|---|
| 灰度 | 全量 | ✓ | **?** | **?** | ✓ | ✓ | **?** |
| 幂等 | cursor | n/a | redis key | db flag | n/a | n/a | n/a |
| 事务 | … | … | … | … | … | … | … |
| 并发 | 分布式锁 | … | … | … | … | … | … |

> **任何一个「?」都是必查项**。矩阵本身是评审产物，必须作为结论的一部分给出。

### Rule 2 — 用户旅程走查（user journey walk）

为 3~5 个典型用户画像写端到端脚本。不看实现细节，只看**用户收到/看到/被阻止了什么**：

- 非灰度城市用户，爽约 2 次 → 收到什么消息？预约会怎样？
- 灰度城市用户，手动释放后再爽约 1 次 → 会怎样？
- 灰度切换瞬间，已有 ACTIVE 黑名单用户 → 会怎样？
- 跨境/跨时区用户（如有）→ 会怎样？
- 异常路径：Bot 发送失败 / 定时任务卡死 / DB 主从延迟 → 用户感知？

许多跨模块不一致只有在这种视角下会暴露。

### Rule 3 — 对「非对称」保持警觉

当看到「规则 R 在模块 X 生效」时，立刻反问：
- R 在 Y、Z、W 也生效吗？
- 如果不生效，设计里是否显式声明了？声明得到产品确认了吗？

**操作方式**：全局 grep 关键词（如「灰度 / gray / switchStatus / grayCityIds」），统计出现位置，缺失的模块就是可疑点。

### Rule 4 — 沉默项显式化

复述每个模块时，**主动补一句**：
- 「灰度判断：文档未提及 / 未启用 / 启用」
- 「幂等控制：文档未提及 / 乐观锁 / 唯一键 / Redis SETNX」
- 「监控告警：文档未提及 / 已覆盖 …」

沉默项一旦被强制标注就会显形。这条的本质是把「没写」从黑盒变成白盒。

### Rule 5 — 两遍阅读（理解 + 对抗）

- **第一遍**：建立正向理解，梳理数据流、状态机、模块协作。
- **第二遍**：带着对抗性姿态专门找漏洞：
  - 我是恶意用户会怎么刷？
  - 我是边界用户（新员工首日 / 离职当天 / 跨时区）会踩什么坑？
  - 我是运维新同学凌晨三点处理告警会困惑什么？
  - 我是接手的工程师半年后要改这段逻辑会怎么被坑？

第一遍 + 零散挑错 = 漏洞率高。必须显式做完第二遍。

---

## 三、评审产出物清单（Deliverable Checklist）

一次合格的方案评审应交付：

- [ ] 横切维度矩阵（Rule 1）
- [ ] 至少 3 条用户旅程走查记录（Rule 2）
- [ ] 非对称项审查结果（Rule 3）
- [ ] 沉默项显式列表（Rule 4）
- [ ] 按严重度分级的缺陷清单（🔴 必须 / 🟠 应该 / 🟡 可以）
- [ ] 每个 🔴 缺陷附带至少一个修复方向

## 四、本次评审真实示例（爽约黑名单方案）

**漏掉的缺陷**：Bot 发送逻辑 `noshow_count >= 2 且 punish_bot_sent = 0`，完全没有灰度判断。

**后果**：
- 非灰度城市用户爽约 2 次 → 收到「您被封禁」Bot
- 用户试着预约 → 拦截有灰度 → 正常放行
- 用户投诉：「Bot 说我被封了，为什么我又能约？」

**如果用了 Rule 1 的矩阵**：Bot 列对应「灰度」行会立刻出现「?」，被迫追问。
**如果用了 Rule 2 的走查**：「非灰度城市爽约 2 次」场景会立刻暴露前后矛盾。
**如果用了 Rule 3 的非对称检查**：搜「灰度」发现拦截/取消有、Bot 没有，立刻触发追问。

三条规则任意一条都能命中这个漏洞。之所以漏，是因为三条都没做。

---

## 五、适用范围

本方法论不仅适用于后端方案，也适用于：
- 前端改造方案（维度换成：响应式 / 无障碍 / 暗色模式 / 低端机 / 弱网）
- 数据方案（维度换成：血缘 / 幂等 / 补数 / 回溯 / SLA）
- 运营规则 / 产品策略文档（维度换成：灰度 / 回滚 / 数据归属 / 合规 / 客服话术）

核心不变：**矩阵化 + 走查 + 对抗性阅读**。
